## Objective

external university LMS". Stand up a typed mock (Shell University), wire
EduRAG to it through a real sync pipeline, surface sync history in a UI
page, and prove the contract end-to-end with `npm run build`.

Explicitly out of scope: authentication, multi-tenant, microservices,
breaking the existing CSV ingest path.

## Sync pipeline summary

```
Shell store (JSON)              EduRAG sync layer
─────────────────               ──────────────────────────────
data/shell-university/*.json
        │                       client.fetchEntity (direct|http)
        ▼                                │
serveEntity / data-store.ts              ▼
        │                       assertEnvelope (defensive parse)
        ▼                                │
{ data, meta } envelope                  ▼
                                map* (shell shape → Prisma)
                                         │
                                         ▼
                                upsert per entity (idempotent)
                                         │
                                         ▼
                                deriveAllSummaries + deriveCourseFeatures
                                         │
                                         ▼
                                SyncLog row written
```

Entity order: `courses → students → enrollments → resources → lms-events → grades → advisor-notes`.

Per-entity error handling means a malformed record fails its row but
doesn't abort the run; the SyncLog `status` reflects `success` /
`partial` / `failed` so the UI can render a status chip.

## Data source architecture summary

EduRAG now accepts data from **three** sources, all writing into the
same Prisma schema:

| Source                  | Trigger                              | Logged source           | Status      |
| ----------------------- | ------------------------------------ | ----------------------- | ----------- |
| Synthetic CSV (Phase 2) | `npm run db:ingest`                  | (no SyncLog yet)        | Functional  |
| Shell University        | `npm run sync:university`            | `shell-university`      | Functional  |
| Uploaded CSV            | `/upload` page (planned Phase 6)     | `uploaded`              | Placeholder |

The `/integrations/shell-university` page derives the **current data
source** from the latest `SyncLog` row and falls back to "Synthetic CSV"
if Prisma has data but no log, or "Unknown" if Prisma is empty.

## Assumptions made

1. **Co-located mock, not standalone service.** Per the
   "no microservices" constraint. The contract is shaped so a real LMS
   replacement only requires changing the base URL + adjusting the mapper.
2. **JSON-backed store, not second DB.** Per the
   "no second complicated database" constraint. The mock store is files
   on disk under `data/shell-university/`.
3. **`AdvisorNote` is a new entity introduced by this integration**,
   not retrofitted onto the CSV. It proves the sync pipeline can carry
   data EduRAG didn't previously model.
4. **Direct file read is the default sync transport.** Lets
   `npm run sync:university` work without a dev server up. `--via-http`
   demonstrates the network contract for screenshots / interviews.
5. **External shape deliberately diverges from internal Prisma shape.**
   snake_case, REST conventions, action vocabulary (`viewed`/`submitted`)
   — the mapper does meaningful translation work.
6. **`SyncLog` derives "current data source"** rather than introducing a
   dedicated settings table. Phase 10 (Dataset Modes) will revisit if a
   first-class switch is needed.
7. **Seed reuses the existing CSV** rather than introducing a parallel
   generator. Means CSV and Shell University paths agree on event-level
   data; the divergence is in shape + the added entities (advisor notes,
   names, programs).
8. **`vitest.config.ts` path alias** added once so all future tests can
   transitively import modules that use `@/*`.

## Risks / things to watch in Phase 6

- **Two sources, one schema.** Running `db:ingest` after
  `sync:university` will overwrite some rows in place (catalogue tables
  are upserted; activity logs are wiped per student × course). Phase 6
  + Phase 10 will need clear "current data source" semantics so the
  dashboard never silently lies about what's loaded.
- **`SyncLog.source` strings are stringly-typed.** A future enum
  promotion (Postgres) would be cleaner; the
  `classifyDataSource` helper already centralises the mapping.
- **`--via-http` requires the dev server running in another shell.**
  Document this clearly in the demo script if a recruiter wants to see
  the HTTP path live.
- **Shell University doesn't paginate or support `?since=`.** Fine at
  our 50k-row scale; a real LMS adapter must add a watermark to avoid
  re-fetching the world on every sync.
- **Advisor notes are tone-templated.** Phase 11 will replace with
  advisor-authored notes + outcome tracking.
