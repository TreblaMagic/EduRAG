# Phase 12 — GitHub + Vercel Deployment Plan (EduRAG)

> Status: **planning** (approved 2026-05-29). No code has been written yet.
> Execution will happen in subphases 12A → 12B → 12C → 12D, each in its own
> review-sized commit/PR.

## Context

EduRAG has reached Phase 11 feature-complete (305 passing tests, 16 routes,
optional Python worker). The codebase is portfolio-ready but lives only on
the local filesystem and runs against SQLite. Phase 12's job is to get it
onto **GitHub (public repo)** and **Vercel (live demo)** without breaking
the local-first developer experience and without compromising the honesty
constraints baked into the product.

Approved decisions:

- **Postgres provider:** Neon (free tier; works with the standard `pg`
  driver; portable to Vercel Postgres later since they share an engine).
- **Hosted demo write policy:** all features enabled, with a nightly Vercel
  Cron job that wipes + reseeds the DB so the public URL never drifts into
  a weird state.
- **Subphase ordering:** screenshots / video / CV / LinkedIn happen *after*
  the live demo is up so we capture the deployed UI (not a pre-deploy local
  version).

## 2. Vercel Compatibility Assessment

| Concern | Verdict | Notes |
| ------- | ------- | ----- |
| SQLite as primary DB | ❌ Won't work | Vercel's filesystem is read-only at runtime. Even if we shipped a `dev.db`, it'd be wiped on every cold start and writes would silently fail. |
| Prisma migrations on hot deploy | ⚠️ Needs strategy | `prisma migrate deploy` should run as a Vercel build step. Schema must target Postgres. |
| Runtime FS writes | ❌ One path needs fixing | `dataset-mode/store.ts` is the only offender. Move state to a Prisma `AppSetting` (singleton row) → works on both SQLite and Postgres. |
| Runtime FS reads (Shell University JSON) | ✅ Works if committed | The 7 JSON files generated by `npm run shell:seed` need to ship in the repo (currently gitignored). On Vercel they'll live inside the deployment bundle — read-only access is fine. |
| Upload (`/upload`) | ✅ Works as-is | Parses CSV in memory, writes to Prisma. Already configured with `bodySizeLimit: "20mb"`. **Note:** Vercel Hobby has a 4.5 MB body limit; Pro raises it to 100 MB. Demo CSV is ~5 MB — needs Vercel Pro OR a smaller demo CSV. **Plan recommendation: shrink the seeded demo CSV to ≤ 4 MB for hosted compatibility on Hobby plan.** |
| Generated reports | ✅ Works | Already in-memory; the HTTP endpoint streams the body. No FS writes. |
| Shell University HTTP routes | ✅ Works | Pure read of bundled JSON. Module-mtime cache becomes module-singleton cache (mtime never changes after bundle is sealed). |
| Long-running CLI commands | ❌ Not invocable from browser | CLIs are never reachable from the web. Build-time + cron-time invocation is the right model. |
| `reset:demo` from UI | N/A | Already CLI-only. No UI button exposes it. Stays local-only. |
| Python worker | ❌ Not on Vercel | Functions can't spawn long-running Python subprocesses; cold start + dependency size makes it impractical. Already gracefully falls back to TS baseline via Phase 7 `selectEngine` / Phase 9 `selectPredictionEngine`. **Hosted demo: baseline-only**. |
| Vercel Cron Jobs | ✅ Available on free tier | Up to 2 cron jobs on Hobby; we need 1 (nightly reseed). |
| Next.js 15 + Prisma cold start | ✅ Fine | Standard pattern; first request after cold start can be ~1-2s but acceptable for a demo. |

## 3. Recommended Hosted Architecture

```
                  ┌───────────────────────────────────────────┐
                  │   Vercel (Next.js production)             │
                  │                                            │
                  │   Server actions, route handlers,         │
                  │   API endpoints — all reading/writing      │
                  │   via Prisma over @neondatabase/serverless │
                  │   (HTTP-pooled connection)                 │
                  │                                            │
                  │   Shell University JSON: bundled in repo  │
                  │   AppSetting (dataset mode): Postgres row │
                  └────────────────────┬──────────────────────┘
                                       │
                            DATABASE_URL (Neon pooler URL)
                                       │
                  ┌────────────────────▼──────────────────────┐
                  │   Neon Postgres (free tier)                │
                  │   pre-seeded by build step + nightly cron  │
                  └────────────────────────────────────────────┘

                  ┌───────────────────────────────────────────┐
                  │   GitHub repo (public)                     │
                  │   - main → auto-deploys to Vercel prod    │
                  │   - PRs → preview deploys                  │
                  │   - GitHub Actions: typecheck + tests +    │
                  │     build on every PR                      │
                  └────────────────────────────────────────────┘

  Local mode (unchanged):
    - sqlite stays a supported provider via a tiny conditional in
      schema.prisma — `npm run setup` continues to work end-to-end
    - Python worker, CLI tools, full intervention authoring all work
    - `.env.local` overrides `DATABASE_URL` to `file:./prisma/dev.db`
```

**Single Prisma schema, two providers.** Prisma supports
`provider = env("DATABASE_PROVIDER")` since 5.x — we use `postgresql` on
Vercel and `sqlite` locally without forking the schema. Migrations are
regenerated against Postgres (the old SQLite migration history is replaced
— the data is synthetic, no harm done) and we keep a second
`prisma/migrations-sqlite/` directory for the local SQLite path.

**Alternative considered:** drop SQLite entirely and ship a
`docker-compose.dev.yml` with Postgres for local dev — cleaner, but breaks
the Phase 9 "two-command demo on a fresh clone" promise. **Decision:** keep
SQLite local; Postgres for hosted.

## 4. Database Migration Plan

### 4.1 Schema changes

Update `prisma/schema.prisma`:

```prisma
datasource db {
  provider = env("DATABASE_PROVIDER")  // "sqlite" locally, "postgresql" on Vercel
  url      = env("DATABASE_URL")
}
```

**Note:** Prisma's `provider` field only accepts a literal string OR an env
var when the env var is set at `prisma generate` time. We'll set
`DATABASE_PROVIDER=postgresql` in Vercel and `DATABASE_PROVIDER=sqlite` in
`.env.example` / local `.env`.

### 4.2 New singleton model for dataset mode

```prisma
model AppSetting {
  key       String   @id
  value     String   // JSON-encoded blob; for dataset-mode it's the existing DatasetModeStateFile shape
  updatedAt DateTime @updatedAt
}
```

Replace `src/server/dataset-mode/store.ts` reads/writes with Prisma calls
against `AppSetting` where `key = "dataset-mode"`. Keep the
`normaliseState` function — it just receives parsed-JSON input instead of
file-parsed input. **This is the only runtime FS write in the codebase**;
once it moves to the DB, the app is fully serverless-compatible.

### 4.3 Migrations strategy

- **Hosted (Postgres):** delete `prisma/migrations/`, run `npx prisma
  migrate dev --name initial` against a local Postgres → commits a single
  fresh migration. Vercel runs `npx prisma migrate deploy` at build time.
- **Local (SQLite):** keep `prisma/migrations-sqlite/` as an alternative
  migration directory. Locally, `npx prisma migrate dev` uses the active
  provider env var.
- **Simpler alternative if multi-provider migrations get fiddly:** drop
  SQLite migrations, document `docker compose up` + Postgres as the only
  local dev path. Already shipped a Dockerfile in Phase 9.

**Recommendation:** ship the multi-provider strategy and only fall back to
Postgres-only-local if Prisma fights us. Document both in the README.

### 4.4 Seed strategy

Add `prisma/seed.ts` that:

1. Runs Phase 9's setup pipeline (`buildSetupSteps()` already exists) but
   routed through the live Prisma client.
2. Skips `npm install` / `prisma generate` / `prisma migrate deploy` steps
   (those run as Vercel build steps).
3. Generates synthetic data in memory (no FS write needed — the existing
   Python generator writes a CSV; for Vercel we ship a committed
   `data/raw/sample_lms_data.csv` and read it at seed time).

Register in `package.json`:

```json
"prisma": { "seed": "tsx prisma/seed.ts" }
```

Vercel build command becomes:
`prisma generate && prisma migrate deploy && prisma db seed && next build`.

### 4.5 Environment variables (DB-related)

| Var | Local | Vercel | Notes |
| --- | ----- | ------ | ----- |
| `DATABASE_PROVIDER` | `sqlite` | `postgresql` | Read at `prisma generate` time |
| `DATABASE_URL` | `file:./prisma/dev.db` | `postgres://...neon...` | Neon pooler URL on Vercel |
| `DIRECT_URL` | (unset) | `postgres://...direct...` | Required for Neon migrations |

## 5. Shell University Deployment Plan

**Stays inside `/api/shell-university/*` as Next.js route handlers.**
Splitting it into a separate service was never necessary; the integration
story is about the *shape* of the contract, not the deployment topology.

**Storage choice for hosted demo: commit the seed JSON files to the repo.**

- Run `npm run shell:seed` once locally → produces 7 JSON files in
  `data/shell-university/`.
- Update `.gitignore` to allow these files (`!data/shell-university/*.json`).
- Verify aggregate size (likely < 2 MB) — small enough to commit comfortably.
- The existing `data-store.ts` module continues to read them; mtime cache
  becomes immaterial on a sealed bundle.

**Sync flow on hosted demo:** `npm run sync:university` is CLI-only and
unreachable from the browser. The seed step (Phase 12B) pre-populates the
DB with a Shell University sync result so the
`/integrations/shell-university` page shows live sync history immediately.
**No live sync trigger from the UI** — that would require a long-running
serverless function. Keep it CLI-only.

**Sync history on the cron-reset path:** the nightly reseed wipes and
re-syncs as part of the seed script. Visitors arriving at
`/integrations/shell-university` always see a fresh "synced 12 hours ago"
entry.

## 7. Upload Strategy

- Upload parsing already happens in memory — works on Vercel.
- `bodySizeLimit: "20mb"` configured; Vercel Hobby caps server actions at
  ~4.5 MB. **Action:** shrink the seeded synthetic CSV to ≤ 4 MB OR
  document that uploads above 4 MB only work on Vercel Pro. Recommendation:
  shrink the default CSV (fewer rows or compress events) AND document the
  limit.
- Add a hosted-only safety rail in the commit orchestrator: cap the number
  of rows imported in a single upload to a sensible ceiling (e.g. 50,000)
  so a malicious or accidental huge upload can't fill the free-tier DB.
  Local mode is unaffected.
- The nightly cron reseeds the DB → any uploaded data disappears after at
  most 24 hours. The UI surfaces this expectation on the `/upload` page
  when `DEMO_MODE=hosted`.
- The upload page already has a HonestyNote banner — extend it with a
  "hosted demo: uploads reset nightly" line when in demo mode.

## 8. Python / Advanced Engine Strategy

**Hosted demo: TS baseline only.** Already the graceful-fallback default —
`selectEngine("advanced")` and `selectPredictionEngine("advanced")` already
detect missing Python and degrade with a structured warning. No code
change needed; Vercel simply won't have Python in its runtime.

The advanced-engine path stays fully functional locally for anyone who
wants the DoWhy + causal-learn + sklearn experience. The README setup
section documents this clearly.

**Long-term option (not Phase 12):** containerise the Python worker and
host it on Fly.io or Render with a small HTTP API. The TS-side `runWorker`
would swap `child_process.spawn` for `fetch`. Phase 13+ work — not needed
for the launch.

## 9. CI/CD Plan

### GitHub Actions: `.github/workflows/ci.yml`

Triggered on every PR to `main` and on push to `main`:

| Job | Steps |
| --- | ----- |
| `ci` | checkout → setup-node 20 → `npm ci` → `npx prisma generate` (uses `DATABASE_PROVIDER=sqlite`, in-memory DB URL) → `npx tsc --noEmit` → `npm test` → `npm run build` (build only, not deploy) |

Runs in < 3 minutes; no external services required (SQLite for the
typecheck/test layer; the real Postgres only matters on Vercel). Use
`actions/cache@v4` to cache `~/.npm` and `.next/cache`.

### Vercel deployment

- Connect the GitHub repo to Vercel.
- Production branch: `main`.
- Preview deploys: enabled for all PRs.
- Build command:
  `prisma generate && prisma migrate deploy && prisma db seed && next build`
- Output directory: `.next` (default).
- Install command: default `npm install` (or `npm ci`).
- Node version: 20.

### Vercel Cron

- `vercel.json` defines one cron: `0 3 * * *` → hits `/api/admin/reseed`
  (new internal route). The route is gated by a `CRON_SECRET` header so
  external traffic can't trigger it.
- The reseed route runs `prisma db seed` programmatically: wipe + repopulate.

### Branch strategy

- `main` is always deployable.
- Feature branches → PR → preview deploy → merge to main.
- Tags `v1.0`, `v1.1`, … only when shipping documented milestones.
